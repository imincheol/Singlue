# 프로젝트 규칙 (Project Rules)

이 문서는 프로젝트 개발 시 반드시 준수해야 할 핵심 규칙을 정의합니다.

## 1. 스펙 주도 개발 (Spec-Driven Development)
- **핵심 원칙**: **"스펙을 먼저 업데이트하고 개발을 진행한다."**
- **개발 프로세스**:
  1.  **스펙 정의 (Spec Update)**: 구현하려는 기능이나 변경 사항을 `docs/` 내의 관련 문서(features.md, data_structure.md 등)에 **가장 먼저** 상세히 명시합니다.
  2.  **구현 (Implementation)**: 작성된 스펙 문서를 기준(Source of Truth)으로 코드를 작성합니다.
  3.  **동기화 (Synchronization)**: 개발 과정에서 스펙 변경이 불가피한 경우, 코드를 수정하기 전에 문서를 먼저 수정하거나, 구현 완료 즉시 문서를 최신화하여 코드와 문서의 일관성을 유지합니다.
- **목적**: 코드와 문서의 불일치를 방지하고, 명확한 스펙 정의 후 개발을 진행하여 재작업을 최소화합니다.

## 2. 원본 문서 보존 (Immutable Origin)
- **대상**: `docs/origin/` 디렉토리 내의 모든 파일 (예: `init-prompt.md`)
- **규칙**: **절대 수정 금지**.
- **이유**: 프로젝트의 초기 기획 의도와 히스토리를 보존하기 위함입니다. 변경이 필요한 경우 새로운 문서를 작성하여 덮어쓰거나 보완합니다.

## 3. 언어 및 소통
- **기본 언어**: 한국어 (Korean)
- **코드 주석 및 커밋 메시지**: 한국어 권장 (혹은 명확한 영어).
- **사용자 소통**: 무조건 한국어로 진행.

## 4. 패키지 매니저
- **필수**: **pnpm**
- **강제**: `package.json`의 `preinstall` 스크립트를 통해 `npm`이나 `yarn` 사용을 시스템적으로 차단합니다.
